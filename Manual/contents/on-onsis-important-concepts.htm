All students need OENs

All publicly funded students should have a Ministry assigned Ontario Education Number. All privately funded secondary student attending an inspected Ontario secondary school also require OENs.  When OnSIS collects elemental student data, the OEN is used to identify they student. Private elementary schools report aggregate rather than elemental student data because these students do not have OENs.

Identify potential OnSIS errors/warnings at any time

In preparation for OnSIS submissions, schools are encouraged to run pre-validation reports to identify any errors/warnings that may be problematic when submitting batch files to OnSIS. Use the Pre-validation option located in each submission period to generate these reports.  

 

Educators must be updated in OnSIS

When creating batch files that include the teacher of the class, the teacher’s MEN# in the SIS is used to identify the teacher in the file. Every educator in the SIS must have a MEN# in their staff record. OnSIS also requires the teacher’s position code in the batch file, and this code must match the code in OnSIS.  By updating the educators in OnSIS before getting extracts, the SIS can use the matching position code found in the extract. See Update educators in OnSIS for more info.

 

Get extract files from OnSIS before submitting batch files

Before creating any batch files in the SIS, extract files need to be created in the Ministry’s OnSIS application, and these files need to be saved in edsembli’s file manager. Extracts are used to determine whether to send an add or update instruction in the batch file created by the SIS.  See Get extracts for OnSIS for more info.

 

Setup card  

Each submission period includes a setup card. This card will provide quick access to the file manager location where the OnSIS extracts, batch files, and results files are stored for the submission.  

 

Extract files: these are the CSV files that come from OnSIS. They provide info about the state of the OnSIS data.

 

Batch files:  these are the XML files that the SIS creates. They provide add/update instructions so that OnSIS can update their data to match the data in the SIS. These files are created by the SIS and are uploaded to OnSIS for processing.

 

Results files: these are OUT files that OnSIS creates when the batch files are processed. They provide info about whether the file is processed successfully or with errors/warnings. These files are downloaded from OnSIS and then processed by the SIS to create an easy to read PDF file of the results.

 

Select any of the folder icons to quickly access those files in file manager.

 

 

The setup may also contain options that may influence the OnSIS batch files, so it is good practice to check this option before starting any new submission period. See Setup for more info.

 

Purge check boxes

Purge check boxes appear when creating batch files. They are an important concept to understand, and using them is the only way to send files again using the original extracts. When the box is checked, an instruction is included in the batch file to tell OnSIS to reset specific data in OnSIS back to the state it was in when the submission period was first opened. This ‘purge and prepopulate’ instruction is processed by OnSIS before the contents of the batch file are processed. It does not return ALL the data to its original form, but just the data in the section you are working with.

 

The purge box is generally selected if you are re-submitting a batch file so that the contents of the previously submitted batch file are cleared, and the data is reset for another attempt.

 

 

When the purge box is checked, you may notice the purge box automatically checked for other file options as well.  Depending upon the file type, other data may be impacted and automatically purged.  

 

 

In order to better understand the purge process, it is important to understand the relationship in the way the data is stored in the Ministry’s OnSIS application.  Sometimes you will find references to Parent entries vs Child entries.  Each student in OnSIS typically has a parent entry and multiple child entries.  Unfortunately, the parent/child terminology is a computing terminology and is not related to the student’s family (OnSIS does not collect parental information).  The Parent refers to the student, and the Child refers to different portions of data collected for the student (e.g. class enrolment, special education, second language data, etc are all “child”records). 

 

If you select a purge flag that affects a parent entry, it will automatically affect the child entries as well.  If a student starts your school during a submission period, the batch file will “add the student into OnSIS.  If the student is enrolled in a second language program, a subsequent batch file will add the second language program information into OnSIS.  If you purge the student, the second language data will automatically be removed as well.

 

Additional warnings will appear when these purge check boxes are selected to clearly identify what data would be impacted.

 

 

For elementary and secondary school submissions, there is a check box to “Purge and Pre-populate ALL OnSIS data” which may be checked when creating the Class List batch file.  If this box is selected, the entire submission period is purged and pre-populated, enabling you to restart the entire submission period over again.  Please note that this will remove any changes made in OnSIS, including educator data that may have been updated manually or submitted by another application (e.g. a HR system).

 

 

 

Updating data in OnSIS using the ‘Create/Upload/Process’ cycle

Data in OnSIS is updated through the process of creating batch files in the SIS, uploading those batch files to OnSIS, and then processing the results generated by OnSIS (analyzing the errors/warnings). These 3 basic steps are repeated until all the data is submitted and is accurate. Since files need to be transferred between the SIS and OnSIS, the steps may seem a little more complicated until you are comfortable uploading and downloading files.

 

 

Step 1:  In SIS, create batch file

Select the CREATE button on the cards to create the OnSIS batch files. Submit the data in the order of the cards, and typically, wait until you are finished working with the info on one card before moving to the next card. For example, sending SHSM data (using the Other info option) before the student exists in OnSIS will cause errors.  

 

 

You'll see a similar form for each CREATE option.

 

 

The board has access to the OnSIS functionality. Creating files with multiple schools is recommended when the board does the OnSIS reporting rather than the schools. Even though the file has multiple schools, the OnSIS generated errors/warnings can also be parsed by schools when processing the results file.  If creating the files from the board, the standard ‘Selected schools’ card will appear at the bottom as shown here:

 

 

 

When the board creates the batch files, there is a check box to ‘Create separate file for each selected school’. If this box is checked, it will create PDF files for each school’s PDF report of warnings/errors.  

 

After selecting the appropriate CREATE button, follow these steps to create the file:

 

  1. Check the Create file box beside the desired OnSIS batch file or files to be created. Although multiple batch files may be selected, where possible, one file will be created.
  2. If you have previously submitted this file to OnSIS, select the Purge box for the file(s).
  3. Examine the Notes column. Extract not found will appear if an extract is required but the extract has not been loaded into file manager. Hint: select the folder icon beside ‘location of extract files’ to see the extracts.  
  4. Set other options on the form as needed. Use the Help icon on each page for details on these options.
  5. Select the CREATE BATCH FILE option.

 

Once the report has been submitted to the task queue, you may leave this application or generate other batch files. You will receive notification when your file is ready. The created batch files are XML files, and they will be saved in file manager under Provincial > OnSIS > Batch files > ‘name of submission period’ (e.g. March). Hint: quickly see the generated files by selecting the folder icon beside ‘location of batch files’ on the file creation form.

 

When the batch file is created, a PDF file will also be generated if any potential errors or warnings have been found. These reports provide an opportunity to correct the data in the SIS before submitting the data. After making corrections in the SIS, repeat step 1 (recreate the batch file).

Step 3: In SIS, process results

 

Once a batch file has been submitted to OnSIS, the file will be ‘processed’ by OnSIS. The results of this processing will be saved in an OUT file, and it may contain important info about errors/warnings that should be analyzed. By processing this file in the SIS, a report is generated that clearly provides the results in a concise and easy to read format. If multiple schools were included in the file, separate reports may be optionally created for each school for distribution to the schools for data correction.

  2. From the Location card, open the folder tree to Provincial > OnSIS > Results > October (or the submission period you are working on).  The Files card will show this location along with the files (if any) in this location.

iv.  Fix data if needed

Based on the results returned from OnSIS, make the needed corrections to the data in the SIS. After submitting batch files  

After all the data has been updated in OnSIS, all the errors and warnings have been addressed to the best of your knowledge, you must log into OnSIS and start the OnSIS sign-off process for the submission period.  The school must sign-off and then the board must sign-off the school. The sign-off process will run a validation report to ensure that the data is error free, and will run verification reports that are checked to see that numbers and totals are accurate.  The numbers in these OnSIS generated reports should agree with the numbers generated in corresponding reports in the SIS:

 

Verification reports

More verification reports

 

For more information on the sign-off process in OnSIS, see Sign-off.

 

Once the sign-off process has completed, the data is locked in OnSIS so that it may not be changed.  OnSIS provides a Request Data Correction option, if a submission period needs to be re-opened.  Re-opened submissions will need to go through the complete sign-off process again after the data correction has been made.

 

For most submission periods, the sign-off process represents the end of the OnSIS submission period.  From the Ministry’s perspective, the submission period is done, and the user has nothing further to do until the next OnSIS submission period opens.  

 

The October and March secondary school submission periods have one last step to perform after signing-off. By doing this step immediately after sign-off, the next onSIS submission period will avoid unnecessary errors. See Populate OnSIS class code.
